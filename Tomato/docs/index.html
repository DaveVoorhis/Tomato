Tomato

"Write one to throw away."

  -- Fred Brooks

 

Tomato is an experimental prototype of a Java application development environment. It aims to replace text-based Java programming with a point-and-click visual programming language.

Specialised visual programming languages have proven successful in niche markets or genres, such as National Instruments' LabVIEW, the shader language in Houdini, or ToonTalk. Visual programming has not yet made a significant impact on general application development. While efforts like Sun's BeanBox or the BlueJ project are promising, they are limited in capability. Tomato was created to explore the application of visual programming to general purpose development. To our knowledge, Tomato is the first open source development tool of its kind for the Java platform.

The prototype works sufficiently -- though only barely -- to be used to develop parts of itself. As parts of Tomato are developed in Tomato, the development process itself is being evaluated to find ways to improve Tomato. Thus, Tomato development is shaped by real-world development requirements, rather than adherence to a predetermined abstract design. This process of long-term ad-hoc prototyping1 should either produce a working visual programming system, or leave a trail of empirical evidence to indicate why general purpose visual programming systems aren't viable.

Once the Tomato prototype can handle most development tasks and demonstrate that general purpose visual programming is viable, it will be redesigned from scratch. The next generation Tomato will be developed entirely within the final evolution of the Tomato prototype.

 

Features

  • A visual programming language -- based on an "icons on strings" metaphor -- for creating, testing, and compiling scripts and new Java classes. Java classes may be instantiated and manipulated at design-time, which facilitates early testing, debugging, and productive experimentation. Program execution is displayed with dynamic visualisation -- color changes are used to indicate method invocation, variable assignments, etc., and these occur in real time. The discrete stages of designing, writing, testing, debugging, and compiling code are merged into a unified software modelling process.

    The visual language supports dynamic execution of source code during development. Programs can be assembled from small, proven-working units, each test-run during construction. Such units may even continue to execute while they are assembled into a whole.2 This may represent a uniquely powerful "run early, run always" development methodology.

  • An integrated development environment to manage visual source code and generate Java applications. This is of a relatively conventional design.

  • A collection of useful reuseable Java classes with Java source code, including database-enabled derivations of the standard JFC/Swing widgets. These may be used within Tomato or any other Java development environment.

 

Goals

  • Reduce application development time.
  • Reduce developer effort.
  • Reduce the opportunity for syntax and logic errors.
  • Integrate prototyping, development, productive experimentation, and testing into the development process from the earliest stages.
  • Automate and/or encourage best practices.
  • Encourage the development of reuseable classes.
  • Encourage the use of meta-programming.

 

Limitations

Tomato is a work in progress. The current release is a mostly-working prototype, suitable for experiments and creating simple classes or test scripts, but it is not ready for production.

Bugs are legion and the interface is clumsy. Furthermore, the visual scripting and class authoring facilities are incomplete. It is possible to write useful Java classes, but inner classes are not yet supported, building expressions is laborious, and parameters cannot be defined for new methods and constructors. And so on.

The documentation, including this document, is incomplete.

These and other limitations will be addressed in future versions.

In other words, this tomato ain't ripe. Tomato is being released at this embryonic stage in order to develop interest (or at least curiosity) and solicit feedback on the design. Please don't inundate us with bug reports. We know it's buggy.

 

Requirements

Tomato is a Java application, and should run wherever Java is available. Tomato was developed under Linux, however, and has only been tested briefly under Windows.

You will require a version 1.4.x Java 2 Platform and should have a Java compiler.

See the Download and Installation section below for URLs where you can obtain these.

The Java compiler is optional, but without it you cannot compile new classes; you will only be able to work with existing Java classes.

Download and Installation

  1. Download and install the Sun Java 2 Standard Edition SDK from http://java.sun.com/j2se/1.4.1/download.html. Obtain the SDK, not the JRE. The JRE (Java Runtime Edition) does not include the Java compiler.

  2. (Optional) The Sun Java compiler is slow. You may wish to use the Jikes compiler, which is much faster. Obtain it from http://www-124.ibm.com/developerworks/oss/jikes/. Jikes is just the compiler -- you'll still need the Java 2 Platform.

  3. Download Tomato.zip to a directory of your choice.
    HTTP download: Tomato.zip
    FTP download: Tomato.zip

  4. Unzip the Tomato.zip, including all its subdirectories, into a directory of your choice.

 

Running Tomato

Note: In the following instructions, replace <directory> with the directory you created in Step 4 under Download and Installation.

Windows: Select "Start | Run", enter java <directory>/Tomato

UNIX or Linux: Type cd <directory>; java Tomato

MacOS: I have no idea.

Bugs

So many they aren't listed... Remember, this is a prototype! Once it reaches Alpha stage, we will start logging bugs.

The most annoying bug is that your visual programming source files -- 'Models' in the Tomato parlance -- may occasionally become corrupt when saved. If you're sufficiently masochistic to use Tomato in production, make frequent backups.

 

Taw-MAY-toe or taw-MAH-toe?

Neither. The correct pronunciation is "TAWM-ah-toe", in order to avoid confusion with a popular vegetable3 of the same spelling.
